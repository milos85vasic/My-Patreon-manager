Organization Repository Support Design

Extending My-Patreon-Manager to handle repositories across multiple organizations

Current Architecture

Current Git Provider Implementation
GitHub Provider: ✅ Full support
• User repos (when org="")
• Organization repos (when org specified)
GitLab Provider: ⚠️ Limited
• Only supports organization/group repos
• No user repository fallback
GitFlic Provider: ⚠️ Limited
• Only supports organization repos
• No user repository fallback
GitVerse Provider: ⚠️ Limited
• Only supports organization repos
• No user repository fallback

Key Questions About Organization Support

What specific organization support do you need?

A

Single Organization Scanning

Scan repositories from one organization at a time (current capability with --org flag)

Pros

  • Simple to implement
  • Matches current CLI pattern
  • Less API quota usage

Cons

  • Multiple runs needed for multiple orgs
  • Manual coordination required
B

Multi-Organization Scanning

Scan repositories from multiple organizations in a single sync operation

Pros

  • Single command for all orgs
  • Unified deduplication
  • Better user experience

Cons

  • More complex configuration
  • Higher API quota usage
  • Potential rate limiting issues
C

Automatic Organization Discovery

Automatically discover all organizations the user belongs to and scan them

Pros

  • Zero configuration
  • Automatic updates as orgs change
  • Most comprehensive coverage

Cons

  • Complex API calls for each provider
  • May include unwanted orgs
  • High API quota consumption

Configuration Approach

How should organizations be configured?

Environment Variables

Separate env vars per provider (e.g., GITHUB_ORGS, GITLAB_GROUPS)

Example: GITHUB_ORGS=my-org,other-org

Configuration File

YAML/JSON config file with provider-specific organization lists

Example: config/organizations.yml

CLI Arguments

Multiple --org flags or comma-separated list in sync command

Example: --org github:my-org --org gitlab:my-group
